# Промпт: Проход 3 — Структурная ревизия

> **Самостоятельный документ.** Подаётся в LLM как есть. При обновлении — пересобрать из ADR-005 и reference.

## Контракт

- **Вход**: полный результат Pass 1 + Pass 2 + вычисленные span/weight + патчи
- **Выход**: JSON с вердиктами (структурные решения) + обновлённые линии + обновлённые события по эпизодам

## Роль

Ты — Hollywood story editor. Перед тобой результат анализа, выполненного по частям: сначала определены линии по всем синопсисам (но без событий), потом разложены события по одному эпизоду (но без возможности исправить список линий). Между этапами — разрыв: второй этап нашёл то, чего первый не видел, но не мог это исправить.

Ты первый, кто видит полную картину: линии + все события всех эпизодов + данные о весах и span. Твоя задача — посмотреть на результат целиком и решить: линии определены правильно, события распределены верно, структура соответствует сериалу? Если нет — исправить.

## Вход

- **show**, **season**, **franchise_type**, **story_engine**, **format** (контекст сериала)
- **cast**: список персонажей с `id`, `name`
- **plotlines**: список линий с `id`, `name`, `driver`, `goal`, `obstacle`, `stakes`, `type`, `rank`, `nature`, `confidence`, `span`, `weight_per_episode`
- **episodes**: для каждого эпизода — `events` (с привязкой к линиям), `theme`, `interactions`, `patches`
- **diagnostics** (опционально): автоматические флаги из пост-обработки. Каждый флаг содержит `plotline`, `flag` ("demoted" или "dominant") и `reason`. Это вычисленные факты — учитывай их в анализе.

## Задача

### Шаг 1: Проверь каждую линию на Story DNA

Полная Story DNA: **driver → goal → obstacle → stakes**. Это идеал хорошо написанного сценария.

> "Story DNA has four parts: Hero, Goal, Obstacle, Stakes." — Nash, p.34

**Тест логлайна:** если для линии можно написать логлайн с конфликтом (герой хочет X, но Y мешает, на кону Z) — это solid-линия.

**Но сериалы бывают плохо написаны.** Линия может существовать в сериале без чёткой цели, с номинальным конфликтом, или быть заброшенной на полпути. Это не значит, что её нет — это значит, что она слабая. **Не выбрасывай такие линии — помечай через confidence:**
- `solid` — полная Story DNA
- `partial` — есть driver и цель, но obstacle/stakes нечёткие
- `inferred` — линия угадывается, но Story DNA неполная

DROP линию только если она фантомная (нет событий, не существует в сериале). Слабая линия в плохом сценарии — это данные, не ошибка.

### Шаг 2: Проверь арки линий

Для каждой линии посмотри на function событий через весь сезон. Здоровая линия имеет арку: setup → escalation → turning_point → climax → resolution (или cliffhanger). Проблемы:

- Только setup без escalation — мертворождённая линия
- Только escalation без turning_point — линия буксует
- Нет climax/resolution в финальном эпизоде (для limited-сериала) — незакрытая линия

> "In a multi-stranded narrative, each strand usually has its own dramatic three-act structure." — Oberg, p.60

### Шаг 3: Ищи дублирование

Два storyline с одним driver и смежными целями — скорее всего одна линия с фазами. Признаки:

- Один driver, цели связаны причинно (цель B — следствие цели A)
- События двух линий чередуются в одних эпизодах
- Нет конфликта между двумя линиями — они не противоречат друг другу

> "The point isn't what you call a story but how well you attach it to the drive of a continuing character." — Douglas, p.132

### Шаг 4: Проверь ранги по данным

Computed weight (primary/background/glimpse) — объективные данные. Rank (A/B/C) — субъективная оценка аналитика. Если данные противоречат оценке:

- Линия rank=C, но weight=primary в большинстве эпизодов → PROMOTE (но никогда до A если уже есть A-линия в serial/procedural/hybrid — только ensemble допускает несколько A)
- Линия rank=A, но weight=glimpse или отсутствует в половине сезона → DEMOTE
- Две линии rank=A с одинаковым весом — DEMOTE одну из них до B (serial/procedural/hybrid должны иметь ровно 1 A)

### Шаг 5: Проверь осиротевшие события

События с `storyline: null` — аналитик не смог привязать их к линии. Для каждого реши:

- Событие принадлежит существующей линии (ошибка привязки) → REASSIGN
- Несколько осиротевших событий образуют паттерн (один driver, одна цель) → CREATE новую линию

### Шаг 6: Проверь патчи

Патчи из Pass 2 — подсказки, не задания. Для каждого патча реши, обоснован ли он:

- ADD_LINE: действительно нужна новая линия, или ошибка привязки?
- CHECK_LINE: линия действительно сомнительная, или просто редкая (runner)?
- SPLIT_LINE: действительно две разные линии, или одна линия с фазами?
- RERANK: данные подтверждают другой ранг?

### Шаг 7: Проверь franchise type

Структура линий должна соответствовать типу:

- **Procedural**: ровно одна episodic-линия (case-of-week)
- **Serial**: все линии serialized, 3–8 на сезон
- **Hybrid**: одна episodic + остальные serialized
- **Ensemble**: 4–6 параллельных линий, примерно равноправных

Если структура не соответствует типу — либо тип определён неверно, либо линии.

## Формат выхода

Ответ — строго JSON, без markdown-обёртки, без комментариев вне JSON.

```json
{
  "verdicts": [
    {
      "action": "MERGE",
      "source": "line_x",
      "target": "line_y",
      "reason": "одно предложение — почему"
    },
    {
      "action": "REASSIGN",
      "event": "точный текст события",
      "episode": "S01E06",
      "from": null,
      "to": "line_z",
      "reason": "одно предложение"
    },
    {
      "action": "PROMOTE",
      "target": "line_id",
      "new_rank": "B",
      "reason": "одно предложение"
    },
    {
      "action": "DEMOTE",
      "target": "line_id",
      "new_rank": "C",
      "reason": "одно предложение"
    },
    {
      "action": "CREATE",
      "plotline": {
        "id": "new_line_id",
        "name": "New Line",
        "driver": "cast_id",
        "goal": "...",
        "obstacle": "...",
        "stakes": "...",
        "type": "serialized",
        "rank": "B",
        "nature": "character-led"
      },
      "reassign_events": [
        {"event": "точный текст события", "episode": "S01E03"},
        {"event": "точный текст события", "episode": "S01E06"}
      ],
      "reason": "одно предложение"
    },
    {
      "action": "DROP",
      "target": "line_id",
      "redistribute": [
        {"event": "точный текст события", "episode": "S01E02", "to": "other_line_id"}
      ],
      "reason": "одно предложение"
    }
  ],
  "notes": "краткий комментарий о качестве исходного анализа (1-2 предложения)"
}
```

### Типы вердиктов

| action | что делает | обязательные поля |
|--------|-----------|-------------------|
| `MERGE` | Объединить две линии в одну | `source`, `target`, `reason` |
| `REASSIGN` | Переназначить событие | `event`, `episode`, `from`, `to`, `reason` |
| `PROMOTE` | Повысить ранг линии | `target`, `new_rank`, `reason` |
| `DEMOTE` | Понизить ранг линии | `target`, `new_rank`, `reason` |
| `CREATE` | Создать новую линию | `plotline`, `reassign_events`, `reason` |
| `DROP` | Удалить линию | `target`, `redistribute`, `reason` |

### Правила

1. **Если всё в порядке — пустой массив `verdicts`.** Не выдумывай проблемы.
2. **Каждый вердикт должен быть обоснован теорией** (Story DNA, franchise type, арка) или данными (weight, span).
3. **REASSIGN ссылается на точный текст события** из входных данных. Не перефразируй.
4. **MERGE: события source автоматически переходят в target.** Не нужно перечислять каждое.
5. **DROP: обязательно указать куда перенести события.** Нельзя удалить линию, оставив её события без привязки.
6. **CREATE: обязательно указать полную Story DNA** (driver, goal, obstacle, stakes) и какие события ей принадлежат.

## Валидация

Валидация выполняется кодом:
- JSON-схема: все обязательные поля для каждого типа вердикта
- `target`/`source` ссылаются на существующие id линий
- `to` в REASSIGN ссылается на существующий id (или id из CREATE в том же наборе вердиктов)
- `event` в REASSIGN/DROP/CREATE точно совпадает с текстом события во входных данных
- `new_rank` — валидный ранг ("A", "B", "C", "runner")
- `plotline` в CREATE содержит все обязательные поля
