# Open Source стратегия

## Контекст

Три проекта: books2series, mas4bw, how2pitch. Вопрос — что и как имеет смысл открывать.

## Когда open source работает

Open source имеет смысл когда есть **точки расширения** — API, схемы, плагины — а не просто интерфейс. Иначе люди просто скачают и пользуются, контрибьютить незачем.

Три модели:
1. **Библиотека/пакет** — код который другие встраивают в свои проекты через импорт и API
2. **Инструмент с плагинами** — ядро фиксированное, можно добавлять модули (жанры, языки, схемы)
3. **Датасет/база знаний** — люди дополняют, исправляют, переводят

## Оценка по проектам

### how2pitch — база знаний + визард
- **Как датасет — да.** Люди могут добавлять книги, переводы, карточки
- **Как приложение — нет.** Визард просто скопируют, контрибьютить некуда
- **Риск:** низкий, это не конкурентное преимущество

### mas4bw — анатомия сезона
- **Как библиотека — самый интересный вариант**
- Выделить ядро экстракции в Python-пакет: `pip install mas4bw`
- API: `extract_arcs(episodes, language="ru", genre="drama")` → структурированные арки
- Вход: тексты/синопсисы серий
- Выход: арки с типом (anthology/soap/genre), персонажами, прогрессией по эпизодам
- **Кому нужно:** исследователи (computational narratology), разработчики инструментов для сценаристов, студии
- **Риск:** средний, основан на открытой статье (ICAART 2025)

### books2series — от книги к питчу
- **Как фреймворк — самый сильный кандидат технически**
- Формальные схемы событий, предикаты, entity resolution — то что люди хотят переиспользовать
- **НО:** это главное конкурентное преимущество, открывать рискованно
- **Вариант:** open core — ядро бесплатно, дашборд + интеграции платно

## Бизнес-модель: Open Core

Так работают PostgreSQL, GitLab, Streamlit:
- **Бесплатно:** ядро (экстракция), базовый функционал
- **Платно:** дашборд, интеграция между проектами, кастомные жанры, поддержка, SaaS

## Что нужно для выпуска библиотеки (mas4bw)

1. Выделить ядро экстракции из монолита (отделить от БД, API, фронтенда)
2. Оформить как Python-пакет с чистым API
3. Описать входы/выходы, типы данных
4. README с примерами использования
5. Опубликовать на PyPI + GitHub
6. Выбрать лицензию (MIT — максимально открытая, Apache 2.0 — с защитой патентов)

## Открытые вопросы

- Готовность поддерживать пакет (issues, PR, документация) — это отдельная работа
- Если не готова — просто выложить на GitHub с MIT лицензией без обещания поддержки
- Какой Reddit: r/screenwriting (~1M), r/LanguageTechnology, r/MachineLearning
