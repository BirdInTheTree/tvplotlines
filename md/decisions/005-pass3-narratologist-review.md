# ADR-005: Pass 3 — ревизия нарратолога

## Статус
Проектирование

## Контекст
Pass 2 генерирует патчи (ADD_LINE, CHECK_LINE, SPLIT_LINE, RERANK) — обратную связь о проблемах в списке линий из Pass 1. Сейчас патчи собираются, но не применяются.

Первоначальный план — перезапустить Pass 1 с учётом патчей. Проблемы:
- Дорого: повторная обработка всех синопсисов
- Рискованно: LLM может сломать то, что уже правильно
- Реактивно: работает только с тем, что Pass 2 заметил

## Решение
Отдельный Pass 3 с ролью **нарратолога** — «второе мнение» от другого взгляда.

### Три пасса — три взгляда
- **Pass 0** — генералист (контекст сериала)
- **Pass 1 + Pass 2** — аналитик (линии + события)
- **Pass 3** — критик/нарратолог (ревизия результата)

### Что видит нарратолог
- Линии + каст (Pass 1)
- События по эпизодам (Pass 2)
- Computed span и weight (пост-обработка)
- Патчи из Pass 2 как *подсказки*, не как задания
- Story engine и franchise type для контекста

### Какие вопросы задаёт
Не «проверь на ошибки», а теоретические вопросы:
- У каждой линии есть driver → goal → obstacle → stakes? Если нет — это не storyline, а thread
- Арка полная? setup без escalation — мертворождённая линия
- Два storyline с одним driver и смежными целями — одна линия с фазами?
- Computed weight = "primary", rank = "C" — кто прав?
- Franchise type = hybrid, но нет episodic-линии — что-то не так
- ADD_LINE: реально новая линия, или ошибка привязки события?

### Выход — вердикты (не патчи)
Вердикты достаточно конкретны, чтобы код применил их механически:
- `MERGE(line_x → line_y)` — все события line_x → line_y
- `REASSIGN(event, → line_z)` — точечное переназначение
- `PROMOTE/DEMOTE(line, new_rank)` — смена ранга
- `CREATE(new_line {driver, goal, ...}, events: [...])` — новая линия + какие события ей принадлежат
- `DROP(line, redistribute: {event → line, ...})` — удаление линии с перераспределением событий

### Применение
Код применяет вердикты к линиям и событиям. Без повторного Pass 1 или Pass 2.

RERANK можно вычислить кодом (из весов) до Pass 3 — нарратолог его не касается.

### Ограничения
- Один вызов LLM, один раунд ревизии
- Если после Pass 3 остаются нерешённые вопросы — оставляем как есть с `confidence: "inferred"`

## Метрики качества пайплайна

Две метрики, считаются вместе как один score:

- **Coverage** — доля нарративных событий из синопсиса, попавших хоть в одну линию (`storyline != null`). Считается из выхода Pass 2.
- **Consistency** — одинаковые синопсисы дают одинаковые линии при разных запусках. Измеряется через ARI (см. ниже).

Формула: **score = coverage × consistency**

Мультипликативная, чтобы нельзя было оптимизировать одно за счёт другого. Coverage = 1.0 при consistency = 0.3 → score = 0.3.

### Consistency через ARI (Adjusted Rand Index)

Наша задача — сравнение кластеризаций. Каждый запуск пайплайна — это кластеризация: события распределяются по линиям (кластерам). Названия линий могут отличаться между запусками («belonging» vs «acceptance»), но структура — какие события оказались вместе — сравнима.

**ARI (Adjusted Rand Index)** — стандартная метрика для сравнения двух кластеризаций без учёта названий кластеров. Диапазон: -0.5..1.0, где 1.0 = идентичные кластеризации, 0.0 = случайное совпадение, <0 = хуже случайного. Реализация: `sklearn.metrics.adjusted_rand_score`, одна строчка.

**Источники:**
- Оригинальная работа: Hubert & Arabie, 1985 — «Comparing partitions» (Journal of Classification)
- Применение к LLM: Guven et al., 2024 — «LLM Stability: A detailed analysis with some surprises» (arxiv:2408.04667) — TAR (Total Agreement Rate) для стабильности LLM-выходов
- Wang et al., 2023 — «Self-Consistency Improves Chain of Thought Reasoning» (ICLR, arxiv:2203.11171) — принцип: несколько запусков + majority vote
- Manakul et al., 2023 — «SelfCheckGPT» (EMNLP, arxiv:2303.08896) — консистентность как прокси для фактологичности

### Два уровня измерения consistency

Это не "или-или" — это разные уровни точности за разную цену:

**Уровень 1: Структурная consistency — ARI через сетку (эпизод × персонаж)**
- Якорь: для каждой пары (эпизод, персонаж) — в какой линии он оказался
- Сетка фиксирована между запусками (персонажи и эпизоды не меняются)
- Грубее, но интерпретируемее и бесплатно считается
- Реализация: `sklearn.metrics.adjusted_rand_score`

**Уровень 2: Семантическая consistency — STED + Hungarian matching**
- STED (arxiv 2512.23712, ноябрь 2025) — фреймворк для оценки consistency structured output LLM. Различает *expression variations* ("belonging" vs "acceptance" — это ок) и *semantic drift* (реально разный результат). Robustness 0.981 для парафраз vs наивный DeepDiff 0.805.
- Hungarian matching (scipy.optimize.linear_sum_assignment) — optimal assignment между наборами линий
- Embeddings каждой линии → Hungarian → среднее cosine similarity по matched парам
- Точнее, но дороже (нужны embeddings)

**Стратегия:** начать с уровня 1 (быстро, бесплатно). Подключить уровень 2 когда ARI перестанет различать.

**Почему не другие якоря для ARI:**
- Номера предложений синопсиса — ненадёжно: одно событие = несколько предложений, LLM склеивает
- Embedding matching → ARI — это фактически Hungarian matching с лишним шагом

**Дополнительные метрики (при необходимости):**
- AMI (Adjusted Mutual Information) — информационно-теоретический аналог ARI
- V-Measure — разделяет homogeneity (чистота кластеров) и completeness (полнота)

### Автоматическая оптимизация промптов (по мотивам autoresearch)

Идея из [karpathy/autoresearch](https://github.com/karpathy/autoresearch): AI-агент в цикле модифицирует код → запускает эксперимент → проверяет метрику → оставляет или откатывает → повторяет. Человек пишет `program.md` (инструкции агенту), а не код.

**Адаптация для Plotter:**

| autoresearch | Plotter |
|---|---|
| Модифицируемый файл: `train.py` | Модифицируемые файлы: промпты `pass{0,1,2,3}.md` |
| Метрика: `val_bpb` (ниже = лучше) | Метрика: `score = coverage × consistency(ARI)` (выше = лучше) |
| Один прогон: 5 мин (GPU) | Один прогон: ~3 мин (API) |
| Логирование: `results.tsv` | Логирование: `results.tsv` |
| Цикл: изменил → запустил → измерил → keep/discard | Тот же цикл |

**Структура program.md для Plotter:**
1. Setup — прочитать архитектуру, промпты, snapshot
2. Constraints — менять только промпты, не код пайплайна
3. Metric — `score = coverage × consistency(ARI)`, выше = лучше
4. Loop — изменил промпт → прогнал N раз → вычислил score → keep/discard
5. Logging — TSV: commit, score, coverage, consistency, status, description

**Ограничения:** каждый прогон стоит денег (API). Для consistency нужно N запусков (минимум 3). Один эксперимент = 3×3 мин = ~10 мин + ~$1-2. За ночь ~50 экспериментов, не 100 как у Karpathy.

Pass 3 (нарратолог) при этом — и этап пайплайна, и качественный индикатор: много тяжёлых вердиктов = Pass 1/2 сработали плохо.

## Что нужно проверить
1. Достаточно ли контекста нарратологу без сырых синопсисов? (видит только линии + события + span/weight)
2. Хватит ли одного вызова, или объём данных (все эпизоды, все линии, все события) не влезет в контекст?
3. Формат вердиктов: сможет ли LLM стабильно генерировать структурированные операции (MERGE, REASSIGN, CREATE)?
4. Применение вердиктов кодом: покрывают ли 5 операций все реальные кейсы?
5. Качество: действительно ли «второе мнение» улучшает результат vs просто добавляет шум?
